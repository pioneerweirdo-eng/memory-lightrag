### Critical integration gaps discovered
- OpenClaw built-in OpenAI TTS provider requests `response_format` mp3 or opus (voice-note channels like Telegram require opus). Bridge v1 returns WAV only → not compatible for end-to-end Telegram voice bubble; likely to cause failures or odd playback.
- `tts.convert` gateway RPC ignores provider overrides passed in params; it always calls `textToSpeech({text, cfg: loadConfig(), channel})`.
- Gateway TTS reads config from `cfg.messages.tts.*`, but also relies on per-session prefs file `~/.openclaw/settings/tts.json` for provider selection (`/tts provider ...`).
- Networking issue: gateway may run in a container; setting baseUrl to `127.0.0.1` may not reach bridge. Need bridge bind `0.0.0.0` and baseUrl use reachable host IP (e.g., `172.18.0.3`).
- Attempted to patch config via `config.apply`/`config.patch`; large raw payload via CLI hits shell arg length limits. Workaround used node `spawnSync` with params.
