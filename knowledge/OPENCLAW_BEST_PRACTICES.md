# OpenClaw 最佳实践（实战版）

> 目标：把“能跑”升级为“可验证、可恢复、可维护”。

## 1) 交付方式：从口头完成到证据化交付

每次任务回报都用 **Trust Receipt**：

1. Intent（目标）
2. Action（做了什么）
3. Evidence（证据：文件路径、命令输出、ID、commit）
4. Risk（风险）
5. Reversibility（回滚方式）
6. Next（下一步）

**原则**：没有证据，不算完成。

---

## 2) 配置与发布

- 配置变更前先备份（或保证 git 可回滚）
- 优先最小改动，不做“全局大手术”
- 变更后必须验证：
  - `openclaw status`
  - 对应功能的真实路径验证（UI/接口）
- 避免“写死覆盖”导致多 agent 行为串扰（例如 UI 全局头像与 agent 头像分层）

---

## 3) 插件与扩展

- 插件来源保持单一、可追溯，避免重复来源/冲突加载
- 出现 `plugin not found`、`unknown channel id`：先校验配置与插件实际安装状态再修复
- 出现 `Cannot find module`：优先检查路径、构建产物、运行目录是否一致

### 启动日志判读（避免误踩坑）

- 连续出现多条 `Exec completed (code 0)` 且内容是 `Registered xxx tool`，通常是正常插件注册与重启过程，不等于故障。
- 先分层判断：
  1. **进程层**：`code 0` / 进程是否退出异常；
  2. **注册层**：插件是否被重复加载、来源是否冲突；
  3. **功能层**：目标工具是否可实际调用（以功能验证为准）。
- 看到 warning（如 provenance/install-path 提示）先归类为“风险提示”而非“立即故障”，除非有明确报错链路指向它。
- 先取证再动作：先收集同一时间窗日志与最小复现，再决定是否重启、回滚或改配置。
- 一次只改一个变量（单点变更），每次改动都记录假设、证据、结果，避免盲目连改。
- 对外汇报采用 Trust Receipt（Intent/Action/Evidence/Risk/Reversibility/Next），确保后续可复盘。

---

## 4) 模型与可用性

- 主模型 + 回退链（fallback）必须配置
- 403/429/502 先判定为上游可用性问题，再看本地配置
- 不要重复发送近似无效回复；给出单次可执行下一步

### 文档优先（官方来源先行）

- 涉及 ACP/Codex/Claude 路由时，先读官方文档再改：
  - `/app/docs/cli/acp.md`
  - `/app/docs/providers/openai.md`
  - `/app/extensions/acpx/skills/acp-router/SKILL.md`
- 明确分层：
  - `openclaw.json -> models.providers` 是 OpenClaw 模型层；
  - `acpx codex/claude` 是 ACP 适配器层；
  - 两者不能混为一谈。
- 任何“可用性判断”必须以实测证据为准（至少包含请求 URL + HTTP 状态 +错误摘要），不是基于模型清单推断。

---

## 5) 多 Agent 设计

- “每个 agent 独立身份”与“UI 全局外观”严格分层
- 每个 agent 资源尽量相对其 workspace 管理
- 不在多个位置维护同一事实数据（Single Source of Truth）

---

## 6) UI 开发落地原则（针对 Control UI）

- 先确认可维护源码路径，再开发；避免硬改编译产物
- 状态就近管理，避免引入多余中间状态
- 功能完成必须提供可点击路径与可复现步骤
- 异常不可吞：错误要可见、可定位、可恢复

---

## 7) 安全与凭证

- API Key 不在聊天明文长期暴露，泄露后立即轮换
- 本地凭证文件最小权限（如 `600`）
- 外部 API 仅发往白名单域名（例如 Moltbook 必须 `www.moltbook.com`）

---

## 8) 运行纪律

- 两次同类失败即换路线，避免死磕
- 15 分钟必须有可验证产出（文件、日志、测试结果）
- 先止血、再优化
- 对“明确且单点”的问题：**直接做最小修复**，不要先铺一堆泛化“消除方案”；修完再给证据与影响面

---

## 9) 学习闭环（持续）

- 每日沉淀：把新经验追加到当日 memory
- 周期整理：把稳定规律升级到本文件
- 对外互动（如 Moltbook）优先高质量帖子，吸收可执行方法并回灌到实践

---

- [ ] 目标是否可验证？
- [ ] 单一事实源是否明确？
- [ ] 回滚路径是否存在？
- [ ] 风险是否被标注？
- [ ] 完成后是否能给出证据（ID/日志/commit）？
